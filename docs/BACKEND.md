# Бэкенд (Supabase)

Справка по схеме БД, ролям и RLS для следующих сессий/разработчиков. Сама схема живёт в
`supabase/migrations/` (SQL — источник истины), здесь — пересказ словами.

## Таблицы

- **`tours`** — каталог туров. Публичное чтение (`using (true)`). Админ — полный CRUD.
- **`leads`** — заявки с сайта (форма заявки, `BookingWidget`). Публичный INSERT (форму
  может отправить кто угодно, без логина). `user_id` — опционально: если заявку
  отправил залогиненный клиент, `useCreateLead` сам подставляет `auth.uid()`.
  Владелец читает свои заявки, админ — читает и меняет статус всех.
- **`reviews`** — отзывы. Публичное чтение только `is_published = true` (фронт
  дополнительно фильтрует явно в `useReviews`, не полагается только на RLS — см.
  комментарий в файле). Админ видит и модерирует все, включая неопубликованные.
- **`faq`** — вопрос/ответ. Публичное чтение, админ — CRUD.
- **`blog_posts`** — статьи блога. Публичное чтение только `is_published = true` (та же
  логика двойного фильтра, что у reviews). Админ — CRUD.
- **`newsletter_subscribers`** — подписка на рассылку. Публичный INSERT, чтения с
  фронта нет.
- **`profiles`** — публичный профиль поверх `auth.users`. `id` = `auth.users.id`.
  Создаётся автоматически триггером `handle_new_user()` при регистрации (email,
  full_name/phone — из `raw_user_meta_data`, которые передаёт `signUp({ options: {
  data } })`). Колонка `role` (`client` | `admin`) — определяет доступ к `/admin`.
- **`favorites`** — избранные туры клиента (`user_id`, `tour_id`, `unique` пара).
  Полностью приватная таблица — виден и редактируется только своим владельцем.

## Роли и RLS

- Аноним (`anon`) — как и раньше: читает публичный контент, создаёт заявки/подписки.
- `authenticated`, роль `client` (профиль по умолчанию) — читает/обновляет свой
  профиль, свои заявки, свои избранные.
- `authenticated`, роль `admin` — полный CRUD на `tours`/`reviews`/`faq`/`blog_posts`,
  чтение и смена статуса всех `leads`, чтение/обновление всех `profiles`.
- Проверка роли — функция `public.is_admin()` (`security definer`, без рекурсии RLS).
  Используется в политиках `Admins manage *`.

### Как назначить первого админа

1. Человек регистрируется через сайт (`/register`) — получает роль `client`.
2. В Supabase Dashboard → SQL Editor выполнить:
   ```sql
   update public.profiles set role = 'admin' where email = 'you@example.com';
   ```
3. Дальше он видит пункт «CRM» в шапке сайта после перелогина (или обновления
   страницы — профиль в `AuthProvider` кэшируется через TanStack Query, ключ `profile`).

Самостоятельно назначить себе admin через фронт нельзя — политика `Users update own
profile` явно запрещает менять свою же `role`.

## Auth

- Email + пароль, `supabase.auth.signInWithPassword` / `signUp` / `signOut`.
- Подтверждение email — зависит от настроек проекта в Supabase Dashboard (Auth →
  Providers → Email → «Confirm email»). Если включено, после регистрации у клиента нет
  сессии, пока не подтвердит письмо — `RegisterForm` это обрабатывает (`needsEmailConfirmation`).
- Восстановление пароля и смена email — **не реализованы** в этой итерации (осознанно,
  см. `docs/CHANGELOG.md`).

## Env-переменные
